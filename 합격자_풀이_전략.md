# 코딩테스트 합격자의 풀이 전략

> 이 문서는 [Claude](https://claude.ai)에 의해 작성되었습니다.

합격자와 불합격자의 차이는 실력보다 **사고 순서**입니다.
합격자들은 문제를 읽자마자 코드를 치지 않습니다.
손을 멈추고 머릿속에서 먼저 다 풀어냅니다.

---

## 풀이 양식

문제를 풀기 전에 아래 주석 구조를 먼저 채웁니다. 코드는 그 다음입니다.

`# 아이디어`와 `# 풀이 흐름`을 분리하는 것이 핵심입니다.

- **아이디어**: "단조 스택으로 O(N)에 해결" — 한 문장 핵심 요약
- **풀이 흐름**: 1. 스택 초기화 → 2. 왼쪽부터 순회 → 3. 현재보다 작은 탑 제거 → 4. 스택 top이 수신탑

구현이 막혔을 때 코드를 고치는 게 아니라 이 설계로 돌아옵니다.

```python
"""
# 아이디어
(핵심 접근법을 한 문장으로)

# 풀이 흐름
1.
2.
3.

# 시간복잡도
O(?) — 근거

# 자료구조
- 변수명: 역할과 저장 형태

# 엣지케이스
-
-
"""
```

---

## 5단계 사고 루틴

### 1단계 — 제약조건 먼저 읽는다 (문제 설명 전에)

대부분은 문제 설명부터 읽지만, 합격자들은 **N의 범위와 시간 제한**을 먼저 봅니다.

N의 크기가 허용하는 시간복잡도를 먼저 알면, 문제를 읽으면서 바로 알고리즘이 보이기 때문입니다.

```
시간 제한 1~2초 기준 (Python은 3~5배 여유 있게 계산)

N ≤ 20            완전탐색 / 백트래킹          O(2^N, N!)
N ≤ 500           DP                           O(N³)
N ≤ 5,000         DP / 정렬                    O(N²)
N ≤ 100,000       정렬 / 이분탐색 / 힙         O(N log N)
N ≤ 1,000,000     스택 / 큐 / 슬라이딩 윈도우  O(N)
```

---

### 2단계 — 예제를 손으로 직접 따라간다

예제를 "확인"하는 것과 "직접 손으로 실행"하는 것은 다릅니다.

자기가 사람 컴퓨터가 된 것처럼 예제 입력을 한 단계씩 계산합니다.
이 과정에서:

- 문제에서 놓친 조건 발견
- 필요한 자료구조가 자연스럽게 보임
- 반복 패턴이 눈에 들어옴

---

### 3단계 — 유형을 분류하고 알고리즘을 선택한다

키워드와 패턴으로 유형을 분류합니다.

| 키워드 / 패턴 | 알고리즘 |
|--------------|---------|
| 최소 비용으로 연결 | MST (크루스칼 / 프림) |
| 최단 거리 | BFS / 다익스트라 |
| 구간 내 최솟값 / 최댓값 | 슬라이딩 윈도우 / 덱 |
| 모든 경우의 수 탐색 | 백트래킹 |
| 구간 합 | 누적합 / 세그먼트 트리 |
| 스택처럼 쌓이는 구조 | 단조 스택 |
| 순서가 있는 최적 선택 | DP |
| 매 순간 최선 선택으로 전체 최적 보장 | 그리디 |
| 연결 여부 / 컴포넌트 수 | BFS / DFS / Union-Find |

#### 그리디 vs DP — 가장 많이 막히는 분기점

- 탐욕 선택이 항상 전체 최적을 보장하면 → **그리디**
- 이전 결과가 이후 선택에 영향을 주면 → **DP**

---

### 4단계 — 코드 전에 설계를 말로 쓴다

합격자들은 코드를 치기 전에 **풀이를 언어로 먼저 씁니다.**

이 글을 쓰는 과정에서 구현 불가능한 아이디어인지 사전에 검증됩니다.
코드가 막히면 코드를 고치는 게 아니라 이 설계로 돌아옵니다.

---

### 5단계 — 엣지케이스를 설계 단계에서 미리 떠올린다

코드를 다 짜고 나서 엣지케이스를 찾는 게 아니라, **설계 단계에서 미리 목록을 만듭니다.**

```
공통 엣지케이스 체크리스트

□ N = 1 (최솟값)
□ 모든 원소가 동일할 때
□ 입력이 이미 정렬되어 있을 때
□ 음수가 포함될 때
□ 결과가 0이거나 음수일 때
□ 정답이 여러 개일 때
□ 연결되지 않은 그래프 (그래프 문제)
□ 사이클이 있을 때 (그래프 문제)
```

## 실전 풀이 예시

### 예시 1 — BOJ 9012 괄호 (Silver IV)

> N개의 문자열이 주어질 때, 각 문자열이 올바른 괄호열인지 판단하라.

#### 1단계: 제약조건 확인

```
T ≤ 50, 문자열 길이 ≤ 50
→ O(N²)도 여유롭게 통과. 자료구조 선택이 핵심.
```

#### 2단계: 예제 손으로 실행

```
(())()  →  열린 괄호를 만나면 카운트 +1, 닫힌 괄호를 만나면 -1
( → 1
( → 2
) → 1
) → 0
( → 1
) → 0
→ 마지막에 0이면 YES
도중에 음수가 되면 즉시 NO
```

예제를 따라가는 순간 "카운터 하나면 된다"는 사실이 보입니다.

#### 3단계: 유형 판단

"열리고 닫히는 구조" → 스택 또는 카운터

스택까지 필요 없고 카운터만으로 충분하다는 것도 예제 추적에서 바로 보입니다.

#### 4단계: 설계 작성 (코드 전)

```python
"""
# 아이디어
열린 괄호는 +1, 닫힌 괄호는 -1로 카운터를 관리.
도중에 음수가 되거나 마지막에 0이 아니면 NO.

# 풀이 흐름
1. 문자열을 왼쪽부터 순회
2. '(' → cnt += 1
3. ')' → cnt -= 1, cnt < 0이면 즉시 NO 반환
4. 순회 후 cnt == 0이면 YES, 아니면 NO

# 시간복잡도
O(N) — 문자열을 한 번 순회

# 자료구조
- cnt: 열린 괄호 누적 카운터 (int)

# 엣지케이스
- 빈 문자열 → cnt=0, YES (문제 조건 확인 필요)
- ')('처럼 닫힌 괄호가 먼저 올 때 → 도중에 음수 → NO 처리 필요
"""
```

#### 5단계: 구현

```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        s = input().strip()
        cnt = 0
        ok = True
        for c in s:
            if c == '(':
                cnt += 1
            else:
                cnt -= 1
                if cnt < 0:
                    ok = False
                    break
        print("YES" if ok and cnt == 0 else "NO")

if __name__ == "__main__":
    solve()
```

### 예시 2 — BOJ 2493 탑 (Gold V)

> N개의 탑이 왼쪽부터 세워져 있다.
> 각 탑의 꼭대기에서 왼쪽으로 레이저를 쏠 때, 어느 탑이 수신하는지 출력하라.
> 수신하는 탑이 없으면 0을 출력한다.

#### 1단계: 제약조건 확인

```
N ≤ 500,000
→ O(N²) = 2.5 × 10^11 → 명백한 TLE
→ O(N log N) 또는 O(N)이 필요
```

이 한 줄만 봐도 브루트포스는 버립니다.

#### 2단계: 예제 손으로 실행

```
입력: 6 9 5 7 4 (탑 높이)

탑1(6): 왼쪽 없음 → 0
탑2(9): 탑1(6)보다 높음, 아무도 수신 못함 → 0
탑3(5): 탑2(9)가 수신 → 2
탑4(7): 탑3(5) 통과, 탑2(9)가 수신 → 2
탑5(4): 탑4(7)가 수신 → 4

출력: 0 0 2 2 4
```

여기서 패턴이 보입니다.
탑3이 탑2에 막히면 탑4도 탑2에 막힙니다.
탑3(5)은 탑4 이후의 탑들에게 영원히 막힐 수 없습니다. 탑4(7)이 가리니까요.

→ 이미 "뒤에서 볼 때 의미 없는 탑"은 버려도 된다는 관찰

#### 3단계: 유형 판단

```
"스택처럼 쌓이는 구조"
"이미 지나간 것 중 필요 없는 것 제거"
→ 단조 스택 (Monotone Stack)
```

스택에 높이가 감소하는 순서로만 유지합니다.
현재 탑보다 낮은 탑은 이후 탑의 레이저도 막을 수 없으므로 제거합니다.

#### 4단계: 설계 작성 (코드 전)

```python
"""
# 아이디어
단조 스택으로 "아직 수신 가능한 탑 후보"만 유지.
현재 탑보다 낮은 후보들은 스택에서 제거.
스택 top이 현재 탑의 레이저를 수신.

# 풀이 흐름
1. 스택 초기화 (비어있음)
2. 왼쪽부터 각 탑 순회
3. 현재 탑 높이보다 낮거나 같은 탑을 스택에서 제거
4. 스택이 비어있으면 0 출력, 아니면 스택 top의 인덱스 출력
5. 현재 탑을 스택에 push

# 시간복잡도
O(N) — 각 탑은 스택에 최대 1번 push, 1번 pop

# 자료구조
- stack: (높이, 인덱스) 튜플의 리스트
  높이는 단조감소 순서로 유지됨

# 엣지케이스
- N = 1 → 왼쪽 탑이 없으므로 0 출력
- 모든 탑이 오른쪽으로 갈수록 높아지는 경우 → 전부 0
- 모든 탑이 동일한 높이 → 바로 왼쪽 탑이 항상 수신
"""
```

#### 5단계: 구현

```python
import sys
input = sys.stdin.readline

def solve():
    N = int(input())
    heights = list(map(int, input().split()))
    stack = []  # (높이, 인덱스 1-based)
    result = []

    for i, h in enumerate(heights):
        while stack and stack[-1][0] < h:
            stack.pop()
        result.append(0 if not stack else stack[-1][1])
        stack.append((h, i + 1))

    print(*result)

if __name__ == "__main__":
    solve()
```

#### 브루트포스와 비교

```python
# O(N²) — TLE
for i in range(N):
    found = 0
    for j in range(i - 1, -1, -1):
        if heights[j] >= heights[i]:
            found = j + 1
            break
    result.append(found)
```

N=500,000에서 브루트포스는 2.5억 번의 비교가 필요하지만
단조 스택은 50만 번의 push/pop으로 끝납니다.

이 차이가 "설계 단계에서 O(N²)을 버린 이유"를 주석에 남겨야 하는 이유입니다.

## 실전 시간 배분

```
문제 읽기 + 예제 추적    2~3분
유형 판단                1분
설계 (주석 작성)         3~5분
구현                     10~15분
디버깅 + 제출            3~5분
────────────────────────────────
합계                     20~30분 / 문제
```

합격자들은 **설계에 전체 시간의 30~40%를 씁니다.**

시험 중 막혔을 때의 판단 기준:

```
구현 10분 넘게 막힘  →  설계로 돌아가기 (코드 디버깅 X)
브루트포스 나왔는데 시간 초과 예상  →  단조 스택 / DP / 이분탐색 고려
```

## 풀이 습관 체크리스트

매 문제마다 아래 순서를 의식적으로 지킵니다.

```
□ 제약조건 (N, 시간 제한) 먼저 확인했는가?
□ 예제를 손으로 직접 실행해봤는가?
□ 시간복잡도를 역산해서 알고리즘 범위를 좁혔는가?
□ 코드 전에 아이디어 / 풀이 흐름 / 시간복잡도 / 자료구조 / 엣지케이스를 주석으로 썼는가?
□ 틀렸을 때 코드가 아닌 설계부터 다시 검토했는가?
```

## 이 프로젝트에서의 적용

`template.py`의 주석 구조를 **실제로 채워서 커밋**하는 것이 핵심입니다.

풀이 코드만 남기면 사고 과정이 사라지지만,
주석까지 남기면 다음이 가능해집니다.

- 복습 시 "왜 이 자료구조를 썼는지" 즉시 파악
- 틀렸을 때 설계의 어느 단계가 잘못됐는지 역추적
- 포트폴리오로서 "이 사람이 어떻게 생각하는지"가 보임

권장 커밋 양식:

```
[Silver IV] 9012 괄호 — 카운터로 O(N) 해결
[Gold V]   2493 탑   — 단조 스택으로 O(N²)→O(N) 최적화
```

알고리즘과 복잡도 개선 이유를 커밋 메시지에 남기면
이력 자체가 성장 기록이 됩니다.
